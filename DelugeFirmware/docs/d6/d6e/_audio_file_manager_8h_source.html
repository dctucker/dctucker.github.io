42/*
43 * ===================== SD card audio streaming ==================
44 *
45 * Audio streaming (for Samples and AudioClips) from the SD card functions by loading
46 * and caching Clusters of audio data from the SD card. A formatted card will have a
47 * cluster size for the filesystem - often 32kB, but it could be as small as 4kB, or even smaller maybe?
48 * The Deluge deals in these Clusters, whatever size they may be for the card, which makes
49 * sense because one Cluster always exists in one physical place on the SD card (or any disk),
50 * so may be easily loaded in one operation by DMA. Whereas consecutive clusters making up an
51 * (audio) file are often placed in completely different physical locations.
52 *
53 * For a Sample associated with a Sound or AudioClip, the Deluge keeps the first two Clusters of that file
54 * (from its set start-point and subject to reversing) permanently loaded in RAM, so playback of the
55 * Sample may begin instantly when the Sound or AudioClip is played. And if the Sample has a loop-start point,
56 * it keeps the first two Clusters from that point permanently loaded too.
57 *
58 * Then as the Sample plays, the currently-playing Cluster and the next one are kept loaded in RAM.
59 * Or rather, as soon as the “play-head” enters a new Cluster, the Deluge immediately enqueues
60 * the following Cluster to be loaded from the card ASAP.
61 *
62 * And then also, loaded Clusters remain loaded/cached in RAM for as long as possible while that RAM
63 * isn’t needed for something more important, so they may be played again without having to reload
64 * them from the card. Details on that process below.
65 *
66 * Quick note - Cluster objects are also used (in RAM) to store SampleCache data (which caches
67 * Sample data post-repitching or post-pitch-shifting), and “percussive” audio data (“perc” for short)
68 * which is condensed data for use by the time-stretching algorithm. The reason for these types
69 * of data being housed in Cluster objects is largely legacy, but it also is handy because all
70 * Cluster objects are made to be the same size in RAM, so “stealing” one will always make the
71 * right amount of space for another (see below to see what “stealing” means).
72 */
