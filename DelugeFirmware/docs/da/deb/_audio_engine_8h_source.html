45/*
46 * ================== Audio rendering ==================
47 *
48 * The Deluge renders its audio in âwindowsâ (or you could more or less say âbuffersâ),
49 * a certain number of samples long. In fact, the Delugeâs audio output buffer is a circular,
50 * 128-sample one, whose contents are continuously output to the DAC / codec via I2S
51 * (Renesas calls this SSI) at 44,100hz. That buffer is an array called ssiTxBuffer.
52 *
53 * Each time the audio routine, AudioEngine::routine(), is called, the code checks where the
54 * DMA outputting has gotten up to in ssiTxBuffer, considers where its rendering of the previous
55 * âwindowâ had ended at, and so determines how many new samples it may now render and write into
56 * ssiTxBuffer without âgetting ahead of itselfâ.
57 *
58 * This scheme is primarily useful because it regulates CPU load somewhat, for a slight tradeoff
59 * in audio rendering quality (more on that below) by using longer windows for each render. For instance,
60 * if the CPU load is very light (as in, not many sounds playing), rendering a window will happen very fast,
61 * so when the audio routine is finished and is then called again the next time, only say one or two
62 * samples will have been output via DMA to the DAC. This means the next window length is set at just
63 * one or two samples. I.e. lighter CPU load means shorter windows.
64 *
65 * Often the window length will be rounded to a multiple of 4, because various parts of the rendering code
66 * (e.g. oscillator table lookup / interpolation) use Arm NEON optimizations, which are implemented in
67 * a way that happens to perform best working on chunks of 4 samples. Also, and taking precedence over that,
68 * windows will be shortened to always end at the precise audio sample where an event (e.g. note-on)
69 * is to occur on the Delugeâs sequencer.
70 *
71 * The window length has strictly speaking no effect on the output or quality of oscillators
72 * (including sync, FM, ringmod and wavetable), filters, most effects, sample playback, or timings/jitter
73 * from the Delugeâs sequencer. Where it primarily does have a (barely audible) effect is on envelope shapes.
74 * The current âstageâ of the envelope (i.e. A, D, S or R) is only recalculated at the beginning of each window,
75 * so the minimum attack, decay or release time is the length of that window, *unless* that parameter
76 * is in fact set to 0, in which case that stage is skipped. So, you can get a 0ms (well, 1 sample) attack time,
77 * but depending on the window length (which depends on CPU load), your 0.1ms attack time could theoretically
78 * get as long as 2.9ms (128 samples), though it would normally end up quite a bit shorter.
79 *
80 * With envelopes (and LFO position actually) only being recalculated at the start of each âwindowâ,
81 * you might be wondering whether youâll get an audible âzipperâ or stepped effect as the output of these
82 * changes sporadically. The answer is, usually not, because most parameters for which this would be audible
83 * (e.g. definitely anything level/volume related) will linearly interpolate their value change throughout
84 * the window, usually using a variable named something to do with âincrementâ. This is done per parameter,
85 * rather than per modulation source, essentially for ease / performance. Wavetable position and FM amounts
86 * are other important parameters to do this to.
87 *
88 * But for many parameters, the stepping is not audible in the first place, so there is no interpolation.
89 * This includes filter resonance and oscillator / sample-playback pitch. Especially sample-playback
90 * pitch would be very difficult to interpolate this way because the (unchanging) pitch for the window
91 * is used at the start to calculate how many raw audio-samples (i.e. how many bytes) will need to be
92 * read from memory to render the (repitched) audio-sample window.
93 */

95/*
96 * ====================== Audio / CPU performance ======================
97 * A huge number of factors influence the performance of a particular Deluge firmware build.
98 * Subpar performance will usually be noticed in the form of sounds dropping out in songs which
99 * performed better in other firmware versions. As an open source contributer, ensuring optimal performance of any code modifications you make,
100 * and subsequently their builds, will be a big challenge. But donât sweat it too much - if youâve
101 * added some cool features which are useful to you or others, maybe lowered audio performance is a reasonable tradeoff?
102 *
103 * The Deluge codebase, since 2021, has been built with GCC 9.2. I (Rohan) compared this with the other 9.x GCC versions,
104 * some 10.x ones, and the 6.x version(s) that the Deluge had used earlier. Performance differences were
105 * negligible in most cases, and ultimately I settled on GCC 9.2 because it resulted in a built binary which was
106 * smaller by a number of kilobytes compared to other versions. GCC 10.x was particularly bad in this regard.
107 *
108 * The build process includes LTO - this helps performance a fair bit. And normal O2 optimization.
109 * These are both disabled in the HardwareDebug build configuration though, for faster build times and
110 * ease of code debugging. If youâre using live code uploading via a J-link and want to do some tests for
111 * real-world performance, you should enable these for this configuration, at least while doing your tests.
112 *
113 * A few other of the standard compiler optimizations are enabled, like âgc-sections to remove unused code from the build.
114 * Beyond that, Iâve experimented with enabling various of the most advanced GCC optimizations,
115 * but havenât found any that improve overall performance.
116 */
