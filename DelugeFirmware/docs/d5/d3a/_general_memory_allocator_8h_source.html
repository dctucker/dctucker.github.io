29/*
30 * ======================= MEMORY ALLOCATION ========================
31 *
32 * The Deluge codebase uses a custom memory allocation system, largely necessitated by the fact that
33 * the Deluge’s CPU has 3MB ram, plus the Deluge has an external 64MB SDRAM IC, and both of these
34 * need to have dynamic memory allocation as part of the same system.
35 *
36 * The internal RAM on the CPU is a bit faster, so is allocated first when available.
37 * But huge blocks of data like cached Clusters of audio data from the SD card are always
38 * placed on the external RAM IC because they would overwhelm the internal RAM too quickly,
39 * preventing potentially thousands of small objects which need to be accessed all the time
40 * from being placed in that fast internal RAM.
41 *
42 * Various objects or pieces of data remain loaded (cached) in RAM even when they are no longer necessarily needed.
43 * The main example of this is audio data in Clusters, discussed above. The base class for all such
44 * objects is Stealable, and as the name suggests, their memory may usually be “stolen” when needed.
45 *
46 * Most Stealables store a “numReasonsToBeLoaded”, which counts how many “things” are requiring
47 * that object to be retained in RAM. E.g. a Cluster of audio data would have a “reason” to
48 * remain loaded in RAM if it is currently being played back. If that numReasons goes down to 0,
49 * then that Stealable object is usually free to have its memory stolen.
50 *
51 * Stealables which in fact are eligible to be stolen at a given moment are stored in a queue which
52 * prioritises stealing of the audio data which is less likely to be needed, e.g. if it belongs to a
53 * Song that’s no longer loaded. But, to avoid overcomplication, this queue is not adhered to in the
54 * case where a neighbouring region of memory is chosen for allocation (or itself being stolen) when
55 * the allocation requires that the object in question have its memory stolen too in order to make
56 * up a large enough allocation.
57 */
